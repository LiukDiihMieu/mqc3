# Documentation Sources for MQC-mini

Source files for the [MQC-mini user documentation](source/index.md).

## Introduction

The user documentation is written in [MyST Markdown](https://mystmd.org/guide/typography)
(`.md`), one section per file. Executable pages are MyST notebooks: plain `.md`
with `{code-cell}` blocks (the [jupytext](https://jupytext.readthedocs.io/) MyST
format) instead of `.ipynb`. This keeps the sources diff-friendly — outputs are
not stored in the files and are regenerated by executing the notebooks at build
time.

It is built with [Jupyter Book 2](https://next.jupyterbook.org/) (the MyST-MD
engine). Configuration and navigation live in `source/myst.yml`.

To edit a page interactively in Jupyter, jupytext can open the `.md` as a
notebook, or pair it back to `.ipynb` with `jupytext --to ipynb <file>.md`.

## Requirements

- Supported Platforms
  - Linux (Ubuntu 24.04 LTS recommended; WSL2 on Windows 11 is also supported)
  - Windows 11
  - macOS (Apple silicon, ARM64)
- Python: 3.10, 3.11, 3.12, or 3.13

## Building the documentation

1. Install the SDK (the docs execute against it), Jupyter Book and jupytext:

   ```sh
   uv pip install torch --index-url https://download.pytorch.org/whl/cpu
   uv pip install ".[torch,dev]"
   pip install jupyter-book jupytext
   ```

   On its first run, Jupyter Book bootstraps a bundled Node environment
   (via `nodeenv`), so the first build needs network access.

2. Build the static HTML site (output in `docs/source/_build/html`):

   ```sh
   cd docs/source
   ./build.sh
   ```

   `build.sh` starts a local Jupyter server, runs the build with `--execute`
   (the `.md` notebooks store no outputs, so they are executed to regenerate
   figures and results), then stops the server. Extra arguments are passed
   through to `jupyter-book`.

3. View or live-preview the site:

   ```sh
   ./build.sh serve             # serve the built _build/html (http://localhost:8080)
   ./build.sh start --execute   # live preview with hot reload + executed outputs
   ```

   Without `--execute`, pages render with **no** code-cell outputs (the `.md`
   sources do not store outputs); use `--execute` or the static `./build.sh`.

   Before opening a PR (or in CI), validate the build including links:

   ```sh
   ./build.sh check   # build + execute + --check-links --strict
   ```

   A plain build silently keeps unresolved links; `--check-links` is required
   to catch dead internal/external links (and `--strict` makes them fail).

   The pages execute against the **`mqc3`** Jupyter kernel
   (`kernelspec: mqc3` in each file's front matter), which points at the
   `mqc3` conda environment. Register it once with:

   ```sh
   python -m ipykernel install --user --name mqc3 --display-name mqc3
   ```

   > **Why `build.sh` runs its own Jupyter server:** mystmd's self-spawned
   > execution server does not start reliably with the ipykernel version in
   > this environment. `build.sh` works around that by launching a Jupyter
   > server and pointing mystmd at it via `JUPYTER_BASE_URL` / `JUPYTER_TOKEN`.

## API Reference (optional, Sphinx-based)

The auto-generated Python API Reference is **not** part of the Jupyter Book 2
build. The legacy Sphinx setup is kept in place so it can be produced (or the
section re-added) when needed:

- `source/conf.py` — Sphinx configuration (autodoc, autosummary, napoleon, …)
- `Makefile` / `make.bat` — Sphinx build entry points
- `source/reference/` — `.rst` stubs generated by `sphinx-apidoc` (git-ignored)

To regenerate it with Sphinx (requires a Sphinx < 9 environment):

```sh
cd docs
sphinx-apidoc -e -M -f -o source/reference/ ../src/mqc3/
make html            # output in docs/build/html
```

### Re-adding the API Reference under Jupyter Book 2 (research notes)

As of mid-2026 there is **no mature native way** to auto-generate a Python API
reference in Jupyter Book 2 / mystmd. The mystmd engine is Node-based and does
**not** run Sphinx, so the JB1-era advice ("add `sphinx.ext.autodoc` to the
config") does not apply here. Native support is still being tracked upstream
(mystmd issue #1259) and is not first-class yet.

Practical options when the API Reference is wanted again:

1. **Separate Sphinx sub-build (recommended).** Keep `conf.py` and build only
   the API pages with Sphinx, publishing them under a sub-path (e.g. `/api`),
   and link to them from the Jupyter Book site (plain links or intersphinx).
   Prefer [`sphinx-autodoc2`](https://myst-parser.readthedocs.io/en/latest/syntax/code_and_apis.html)
   over classic `sphinx.ext.autodoc`: it analyses the source **statically**
   (no need to import the package or install torch), supports MyST docstrings,
   and offers `auto_mode` (whole-package pages) plus the `{autodoc2-object}`
   directive with `render_plugin = "myst"`.
2. **Wait for native mystmd support** — track
   [mystmd#1259](https://github.com/jupyter-book/mystmd/issues/1259); collapse
   to a single build once it lands.
3. **Standalone generators** (`pdoc`, `mkdocstrings`) can render an API site
   from docstrings, but they do not integrate with the Jupyter Book build.

References: mystmd#1259; MyST Parser "Source code and APIs"
(<https://myst-parser.readthedocs.io/en/latest/syntax/code_and_apis.html>);
Jupyter Book FAQ (<https://jupyterbook.org/latest/resources/faq/>).

## Project layout

```text
docs/
├── Makefile / make.bat    # legacy Sphinx build (API Reference)
└── source/
    ├── myst.yml           # Jupyter Book 2 config + navigation (toc)
    ├── build.sh           # build helper (starts a Jupyter server, executes)
    ├── *.md               # documentation pages (MyST; some are notebooks)
    ├── references.bib     # bibliography
    ├── conf.py            # legacy Sphinx config (API Reference)
    ├── _images/           # image files
    └── reference/         # generated .rst API stubs (git-ignored)
```

- `source/` : Source files for the user documentation.
  - `myst.yml` : Jupyter Book 2 configuration and table of contents.
  - `build.sh` : Build helper that runs `jupyter-book` with notebook execution.
  - `*.md` : Documentation pages in MyST Markdown; pages with `{code-cell}`
    blocks are executed at build time.
  - `_images/` : Image files.
  - `reference/` : Generated `.rst` API stubs (used only by the Sphinx build).
